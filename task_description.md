# 工业级 RL 环境缺陷修复与自动化评测基准构建 — 任务描述

## 1. 项目概述

### 1.1 项目名称
工业级 RL 环境缺陷修复与自动化评测基准构建

### 1.2 项目背景
为精准评估 Claude、GPT-4o 等顶尖大模型在强化学习工程中的综合能力，需要构建一套高难度、全自动化的 RL 环境与代码评测数据集。现有评测集多侧重算法理论或简单游戏，缺乏对真实 RL 开发场景中逻辑缺陷、环境漏洞、交互异常的深度考察，难以区分模型在实际工程落地中的强弱差异。

本项目旨在填补这一空白，设计贴合工业级 RL 开发流程的测试考题，覆盖环境搭建、代码调试、训练逻辑修复、自动化验证全链路。

### 1.3 项目目标
- 设计 4 个梯度难度的真实工程场景 RL 任务，每个任务包含精心构造的代码缺陷
- 构建基于 Docker 的可复现运行环境
- 编写全自动化黑盒测试脚本，实现一键验证
- 提供完整的修复方案与纯英文任务描述文档

## 2. 核心交付物

### 2.1 RL-Task（强化学习任务代码）
每个任务包含：
- **有缺陷的 RL 环境代码**：可正常执行但包含逻辑缺陷、环境漏洞、交互异常
- **标准化修复方案**：缺陷修复后的完整代码

### 2.2 Auto-Verify（自动化验证系统）
- Docker 容器化运行环境
- 全自动化黑盒测试脚本
- 一键运行、一键验证能力

## 3. 四个梯度任务

### Task 1: Inventory Management（库存管理）— 难度 ★☆☆☆☆

**场景描述**
单产品定期盘点库存系统。Agent 在每个周期初查看当前库存、在途订单、需求预测等信息，决定订购多少单位产品。目标是最小化持有成本与缺货惩罚的总和。

**缺陷设计**
| 缺陷 | 类型 | 说明 |
|------|------|------|
| 奖励符号反转 | 奖励函数错误 | 奖励返回正数成本值，Agent 学会最大化订购量而非最小化成本 |
| 库存计算错误 | 状态空间错误 | 计算期末库存时未减去当期需求，库存永远不会减少 |
| 永不终止 | 终止条件缺失 | 未设置最大步数限制，episode 永不到达终止状态 |

**预期修复难度**：低。缺陷较明显，符号错误和基本逻辑错误，适合作为入门级考察。

---

### Task 2: Job Shop Scheduling（作业调度）— 难度 ★★☆☆☆

**场景描述**
多作业多机器调度问题。有 N 个作业需要在 M 台机器上加工，每个作业包含多道工序，每道工序指定了所需的机器和处理时间。Agent 需要在每个决策点选择下一个要处理的作业，目标是最小化总完工时间（makespan）。

**缺陷设计**
| 缺陷 | 类型 | 说明 |
|------|------|------|
| 动作屏蔽缺失 | 动作空间错误 | 未屏蔽已完成的作业，Agent 可反复选择无效作业，浪费决策步数 |
| 提前终止 | 终止条件错误 | 步数计数器从 0 开始但在错误时机检查，导致 episode 提前一步结束 |
| 奖励归一化爆炸 | 奖励函数错误 | 奖励归一化时分母接近 0，导致数值溢出和训练不稳定 |

**预期修复难度**：中。需要理解调度逻辑和动作屏蔽机制。

---

### Task 3: Ad Bidding / Campaign Optimization（广告竞价优化）— 难度 ★★★☆☆

**场景描述**
多广告系列预算分配系统。Agent 需要将有限的总预算在多个广告渠道/系列之间动态分配，在每个决策周期决定各渠道的投入金额，以最大化总转化数或 ROI。环境模拟了不同渠道的需求波动、边际效益递减等真实特性。

**缺陷设计**
| 缺陷 | 类型 | 说明 |
|------|------|------|
| 预算泄漏 | 转移动力学漏洞 | 未强制执行预算约束，Agent 可以超支，各渠道花费总和超过总预算 |
| 状态缺失 | 状态空间错误 | 观测空间中缺少剩余预算信息，Agent 无法做出知情的分配决策 |
| 非平稳奖励未归一化 | 奖励函数错误 | 需求随时间波动导致奖励尺度剧烈变化，训练崩溃 |
| 种子未固定 | 不可复现 | 随机种子未被固定，每次运行结果不一致，无法进行可重复调试 |

**预期修复难度**：中高。涉及对预算约束机制、状态空间设计和奖励归一化的综合理解。

---

### Task 4: Cache Replacement Policy（缓存替换策略）— 难度 ★★★★☆

**场景描述**
CDN 内容分发网络的缓存替换策略学习。缓存空间有限，当新的内容请求到达且缓存已满时，Agent 需要决定驱逐哪个已缓存的内容。目标是最大化缓存命中率。环境模拟了真实的内容访问模式，包括热点集中、时间局部性、频率多样性等特征。

**缺陷设计**
| 缺陷 | 类型 | 说明 |
|------|------|------|
| 状态缺失时间特征 | 状态空间错误 | 缺少访问频率、最近访问时间等关键时间特征，状态信息不足以做出最优决策 |
| 延迟奖励累加错误 | 奖励函数错误 | 缓存命中/未命中的延迟奖励在多步之间错误累加，奖励信号与动作的因果关系混乱 |
| 内存泄漏 | 性能陷阱 | step() 函数中的历史记录列表持续增长，随 episode 变长性能急剧下降 |
| 终止条件变量错误 | 终止条件错误 | 终止判断使用了错误的变量引用，episode 永不到达自然终止点 |

**预期修复难度**：高。缺陷较为隐蔽，涉及状态表征设计、时序信用分配和工程性能优化等多个维度。

## 4. 技术栈要求

| 组件 | 规格 |
|------|------|
| 语言 | Python 3.10+ |
| RL 接口框架 | Gymnasium >= 0.29 |
| 深度学习框架 | PyTorch >= 2.0 |
| 科学计算 | NumPy |
| 验证训练 | Stable-Baselines3（可选） |
| 容器化 | Docker，基于 python:3.10-slim |
| 测试框架 | pytest |

## 5. 自动化验证要求

每个任务必须提供 `solution/solve.sh` 一键验证脚本，实现：

1. 安装依赖（或复用全局环境）
2. 运行黑盒测试用例（pytest），覆盖：
   - 动作空间 / 观测空间的维度与类型一致性
   - reset() / step() 接口契约（符合 Gymnasium Env 规范）
   - 奖励范围与终止条件正确性
   - 至少 100 个 episode 的无崩溃运行
   - 随机种子可重复性验证
3. 可选：运行最短训练流程，输出平均回报达到预设基线
4. 退出码 0 表示全部通过，退出码 1 表示存在失败

## 6. 验收标准

- [ ] Docker 镜像构建成功
- [ ] 每个任务 `bash solution/solve.sh` 在 5 分钟内完成所有测试并返回退出码 0
- [ ] 原始问题代码在相同测试下出现至少一项断言失败或训练指标异常
- [ ] 环境接口完全兼容 Gymnasium 最新稳定版
- [ ] 纯英文 TASK.md 文档流畅无歧义
- [ ] 随机种子可控，多次运行结果一致
- [ ] Docker 镜像大小 < 2GB

## 7. 约束条件

- 问题代码不可引入恶意代码或无限循环导致 Docker 崩溃
- 场景难度对标工业级 RL 开发痛点，避免纯学术玩具示例
- 所有测试必须为黑盒式，仅通过 Gymnasium API 交互，不依赖内部实现细节
- 依赖版本需锁定，保证环境可复现
